# MOSAIC 学習型 CPU 研究開発 README

## 1. プロジェクト概要

本プロジェクトは、MOSAIC の CPU を将来的に機械学習ベースで強化するための研究開発用プロジェクトである。現時点では完成済みの学習システムを作ることではなく、学習型 CPU に発展できる土台を Python で整備し、以後の設計判断と実装方針の基準を README として明文化することを目的とする。

通常の手組み CPU は、ルール理解や評価関数の調整を人手で進めやすい一方で、探索やヒューリスティクスの改善が属人的になりやすく、強化の方向性が固定化しやすい。これに対し学習型 CPU は、既存の強い CPU の挙動を模倣したり、自己対戦から新たな方策を獲得したりできる可能性があるため、中長期的な性能向上の余地が大きいと考えている。

本 README は、構想段階の研究開発メモ兼プロジェクト仕様書の初版として扱う。今後の実装や設計判断は、原則として本 README を SSOT として進める。

ルートの `README.md` は、プロジェクト全体の方針、現状、ロードマップを確認するための入口として扱う。詳細仕様や設計ノートは `docs/` 配下に段階的に切り出し、本 README を SSOT の入口として参照する構成を前提とする。

## 2. 背景と前提

### MOSAIC の性質

- 対象ゲームは MOSAIC である。
- 現時点では、学習器より先にゲームエンジン側の整備が必要である。
- ルール詳細や盤面仕様のうち、まだ文書化されていないものは未確定として扱い、README では勝手に固定しない。

### 棋譜データ不足の課題

- 人間の大量棋譜は、存在しないか、存在しても十分量を安定して集められない前提で考える。
- そのため、一般的な教師あり学習を人間棋譜中心で始める方針は取りにくい。
- 初期データは、既存 CPU や自己対戦によって生成する前提で設計する。

### kobalab CPU を活用する発想

- 既存の強めの CPU ロジックとして kobalab の CPU がある。
- まずはこれを教師役、比較対象、評価基準の 1 つとして活用する。
- 初期段階では、kobalab CPU の意思決定を観測し、その着手を学習対象にする模倣学習が最も着手しやすい候補である。
- あわせて、kobalab vs kobalab の自己対戦により、局面と着手のペアを機械的に生成する方針を想定する。

### ローカル環境から始める理由

- 開発はローカルマシンから開始する。
- 利用可能な GPU は NVIDIA GeForce RTX 4060 Laptop GPU（VRAM 8GB）である。
- ただし、初期フェーズでは重い GPU 学習よりも、Python でゲームエンジン、データ生成、評価の基盤を整えることを優先する。
- ローカル環境から始めることで、実験サイクルを短くし、仕様の詰めと設計の見直しを素早く回せる。

## 3. 現時点の開発方針

現時点の方針は、いきなり学習モデルを作り込むことではなく、学習を成立させるための前提基盤を順に整えることである。

- まずは Python 版の MOSAIC エンジンを整備する。
- その上で、CPU 同士を対戦させてデータを生成できる基盤を作る。
- 初期の学習は模倣学習寄りで開始し、後続フェーズで自己対戦や強化学習を検討する。
- GPU は利用可能だが、初期段階の主目的は学習速度ではなく、正しい状態表現、合法手生成、再現可能なデータ生成、評価の仕組みを整えることである。

このため、当面は「学習方式の最適化」よりも「ゲームエンジンと実験基盤の整備」を優先する。

## 4. 学習アプローチ候補

現時点では、学習方式を 1 つに断定しない。以下を候補として整理し、段階的に検証する。

| アプローチ | 現時点での位置づけ | 主な利点 | 主な懸念 |
| --- | --- | --- | --- |
| 模倣学習 | 第一候補 | kobalab CPU を教師役として使いやすい。初期データを作りやすい。 | 教師の癖を強く引き継ぐ可能性がある。 |
| 自己対戦 | 中期候補 | 人間棋譜が少なくてもデータを増やせる。方策改善の足場になる。 | 初期方策が弱いと低品質データが増えやすい。 |
| 強化学習 | 将来候補 | 勝敗や報酬を直接最適化できる可能性がある。 | 報酬設計、探索、安定学習の難易度が高い。 |
| AlphaZero 風拡張 | 長期候補 | 自己対戦と探索を組み合わせた強力な枠組みに発展可能。 | エンジン、探索、学習基盤の完成度が前提になる。 |

### 模倣学習

- kobalab CPU の着手を教師信号として使う想定である。
- 初期段階で最も現実的な入口であり、ベースラインモデル作成に向いている。
- まずは「強い着手を学ぶ」というより、「局面表現から合法手付きの予測パイプラインを成立させる」ための方式として位置づける。

### 自己対戦

- kobalab vs kobalab、または将来的には学習済みモデル同士の対戦でデータを作る想定である。
- 人間棋譜不足を補うための重要な選択肢である。
- ただし、どの自己対戦データをどう採用するかは未確定であり、初期段階で過度に複雑化しないようにする。

### 強化学習

- 勝敗や中間報酬をもとに方策改善する可能性を持つ。
- ただし、本プロジェクトの初期段階では優先度を上げない。
- エンジンの正しさ、評価環境、ベースラインの確立後に検討する。

### AlphaZero 風への将来拡張

- 方策と価値の同時学習、探索との組み合わせ、自己対戦ループなどに発展する可能性はある。
- 現時点では構想レベルに留め、初版 README では実装前提にしない。
- 将来拡張の余地を残すため、初期設計では状態表現、合法手、対戦ループ、評価スクリプトを分離しておく。

## 5. まず必要な技術要素

初期フェーズで必要になる技術要素は以下の通りである。いずれも、現時点では「必要である」という整理であり、実装済みではない。

| 技術要素 | 役割 | 現時点の扱い |
| --- | --- | --- |
| Python ゲームエンジン | 状態遷移、対戦進行、検証の中核 | 最優先で整備する |
| 盤面表現 | 学習入力と内部状態表現の基礎 | 未確定。別紙で設計する |
| 合法手生成 | 学習・推論・探索の前提 | エンジン整備と同時に必要 |
| 終局判定 | データ生成と評価戦に必須 | エンジン整備と同時に必要 |
| kobalab ラッパー | 教師役・対戦相手として接続するための層 | 接続方式は未確定 |
| データ保存形式 | 棋譜、局面、着手、結果の保存 | 早期に決める必要がある |
| 学習スクリプト | 模倣学習などの学習実験用 | エンジン後、早い段階で着手 |
| 評価戦スクリプト | 各プレイヤー同士の比較検証 | 学習スクリプトと並行して必要 |
| GPU 利用基盤 | 学習高速化のための実行環境 | 初期優先度は高くない |

### GPU 利用の位置づけ

- GPU は使えるが、初期段階で前提にしすぎない。
- 小規模なモデル検証や学習速度改善には有効だが、設計不備やルール実装ミスは GPU では解決しない。
- まずは CPU でも動くデータ生成・検証パイプラインを作り、その後に必要に応じて GPU 利用を強める。

## 6. 想定ディレクトリ構成案

以下は現時点の案であり、確定ではない。実装の進行に応じて変更する可能性がある。

| ディレクトリ | 想定役割 |

| `engine` | MOSAIC のゲームロジック、状態遷移、合法手生成、終局判定 |

- `datasets` は生データと加工済みデータを分ける可能性があるが、現時点では未確定である。
- `training` と `evaluation` は、後から試行錯誤しやすいようにスクリプト依存を整理しておく。
- ルートの `README.md` を入口とし、詳細仕様は `docs/` 配下へ段階的に分離していく。

## 7. 段階的な開発ステップ

### Phase 0: README / 仕様整理

- 本 README を初版として整理し、当面の共通前提を固定する。
- 未確定事項を明示し、決まっていないものを勝手に実装で埋めない。
- Python エンジン仕様を別紙化する準備を行う。

### Phase 1: Python エンジン

- MOSAIC の状態表現、合法手生成、着手適用、終局判定を Python で実装できる状態を目指す。
- このフェーズの主目的は、学習前提となる正しいゲーム進行基盤の確立である。
- 盤面表現や action 表現の詳細は、このフェーズで設計を詰める。

### Phase 2: kobalab 対戦・棋譜生成

- kobalab CPU と接続し、対戦を自動実行できる状態を目指す。
- kobalab vs kobalab、または Python 側プレイヤーとの対戦により、局面データを蓄積できるようにする。
- ここで、棋譜・局面・着手・結果の保存形式を実務上使いやすい形に固める。

### Phase 3: 模倣学習

- 生成したデータをもとに、まずは模倣学習のベースラインを作る。
- 目標は最初から高性能モデルを作ることではなく、学習、推論、対戦評価の一連の流れを成立させることである。
- 着手再現率や簡易対戦結果を通じて、最低限の有効性を確認する。

### Phase 4: 評価リーグ

- random、現行 CPU、kobalab、学習済み旧版、新版モデルを比較できる評価リーグを整備する。
- 勝率だけでなく、再現性と安定性を見ながら改善サイクルを回す。
- 評価条件の固定方法やシード管理も、この段階で明確化する。

### Phase 5: 自己対戦 / 強化学習検討

- 模倣学習ベースラインの結果を踏まえ、自己対戦や強化学習への拡張可否を判断する。
- この段階で初めて、探索付き学習や AlphaZero 風の構成を本格検討する。
- ここでの判断材料は、性能だけでなく、実装コスト、再現性、学習安定性も含む。

## 8. 評価方針

本プロジェクトでは、「モデルが動いたこと」ではなく、「比較可能な形で前進したこと」を評価する。

### 何をもって前進とするか

- 正しいルールに基づいて対戦やデータ生成が再現可能に動くこと
- 比較対象に対して、一定の指標で改善または差分が確認できること
- 学習や評価の結果を、後から追跡・再実行できること

### 比較対象

- random
- 現行 CPU
- kobalab
- 学習済み旧版
- 学習中の新版

### 見る指標

- 勝率
- 着手再現率
- 対戦結果の安定性
- 同一条件での再現性
- 不正手や異常終了の有無

勝率だけでは判断しない。模倣学習の初期段階では、教師の着手をどの程度再現できるか、対戦挙動が破綻しないか、といった観点も同等に重要とする。

## 9. 想定リスク・注意点

- kobalab の癖を学習しすぎる可能性がある。
- 生成データが特定の局面や戦型に偏る可能性がある。
- 盤面表現を誤ると、学習以前にデータ全体が不正確になる。
- 合法手マスクが不十分だと、学習時も推論時も評価が歪む。
- GPU を使えることと、良い設計ができていることは別問題である。
- 初期段階では、学習器の複雑さよりも、エンジンの正しさとデータ品質の方が重要である。
- 未確定仕様を暗黙の実装で固定してしまうと、後工程で大きな手戻りになる。
