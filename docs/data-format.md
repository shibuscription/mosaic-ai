# MOSAIC データ形式仕様書 v0

## 1. 文書の目的

本書は、MOSAIC 学習型 CPU 研究開発プロジェクトで扱うデータの種類、役割、単位、保存方針を整理するための初期仕様書である。目的は、今後の実装や運用の中で対局ログ、棋譜、局面、学習用サンプル、評価結果の形式を場当たり的に決めることを避け、最低限の共通前提を先に持つことである。

ルートの `README.md` は全体方針とロードマップの入口、本書はデータ設計の詳細文書、`docs/engine-spec.md` はエンジン仕様の詳細文書という役割分担を前提とする。本書は、特にエンジンのシリアライズ方針、学習データ生成、評価再現性に関わるデータ面の設計を扱う。

現時点では実装前の仕様整理文書であり、最終形式をすべて確定するものではない。未確定事項は未確定のまま残しつつ、少なくとも実装時に迷いにくい単位と責務の切り分けを示す。

## 2. このプロジェクトで扱うデータの種類

本プロジェクトで主要に扱うデータは以下の通りである。

| データ種別 | 主な用途 | 主に必要になるフェーズ |
| --- | --- | --- |
| 対局メタデータ | 対局条件、使用プレイヤー、生成経路、再現条件の記録 | Phase 2 以降 |
| 棋譜 | 対局の流れ、手順列、勝敗結果の保存 | Phase 2 以降 |
| 局面スナップショット | 特定局面の再現、デバッグ、回帰テスト | Phase 1 以降 |
| 学習用サンプル | policy 学習、将来の value 学習、教師データ生成 | Phase 3 以降 |
| 評価戦結果 | モデル間比較、ベンチマーク、進捗確認 | Phase 4 以降 |
| 推論ログ / モデル評価ログ | モデル出力確認、挙動解析、異常調査 | 必要に応じて |

### 対局メタデータ

- 対局そのものの中身ではなく、対局をどういう条件で実行したかを記録する。
- 使用エンジン、プレイヤー種別、seed、生成日時、ルール設定などを含む候補である。
- データ生成の再現性、比較可能性、検証容易性のために重要である。

### 棋譜

- 対局の流れを保存する中核データである。
- 1 局を再生・再現できることを主目的とする。
- 学習用サンプル生成の元データにもなりうる。

### 局面スナップショット

- 特定時点の state を切り出したデータである。
- 不具合再現、回帰テスト、途中局面の確認、局面共有に向く。
- 棋譜全体より軽く、単独で再利用しやすい。

### 学習用サンプル

- モデルに与える入力と教師信号を、局面単位またはサンプル単位で切り出したデータである。
- 模倣学習の初期段階では特に重要である。
- 対局ログや棋譜とは別物として管理する方が扱いやすい。

### 評価戦結果

- プレイヤー同士の比較結果を集約したデータである。
- 個々の対局ログとは別に、リーグや比較条件の結果を集計して保存する。
- README の評価方針に基づく比較可能性の土台になる。

### 推論ログ / モデル評価ログ

- 必須ではないが、モデルの出力や意思決定を確認したい場合に有用である。
- 予測分布、上位候補手、mask、value 出力などを記録する可能性がある。
- 初期段階では常時保存前提にせず、調査用の補助データとして扱う。

## 3. データ単位の考え方

本プロジェクトでは、同じ対局から複数の粒度のデータが生まれる。これらは役割が異なるため、明示的に区別する。

| 単位 | 内容 | 主な用途 |
| --- | --- | --- |
| 1対局単位 | 1 回のゲーム全体 | 棋譜保存、勝敗確認、再現 |
| 1手単位 | 対局中の各 action | 棋譜解析、ログ確認 |
| 1局面単位 | ある時点の state | デバッグ、学習入力生成、回帰テスト |
| 1学習サンプル単位 | 入力局面と教師信号の組 | 学習、検証、統計処理 |

### 1対局単位

- 1 回のゲーム開始から終了までをまとめた単位である。
- プレイヤー情報、手順列、結果、対局条件などを持つ。
- 棋譜の自然な単位である。

### 1手単位

- 対局中の各 action を順番に扱う単位である。
- 対局ログのイベント列や手順列として使いやすい。
- ただし、1手単位だけでは盤面全体を再現するには初期局面やルールバージョンも必要である。

### 1局面単位

- 特定タイミングの state を扱う単位である。
- 同じ対局から複数局面が得られる。
- 学習入力、既知局面テスト、不具合再現に向く。

### 1学習サンプル単位

- 局面に加え、教師手や結果ラベルなどの学習対象情報を束ねた単位である。
- 同一対局から多数生成される。
- 棋譜と学習サンプルは一致するとは限らない。

### 棋譜と学習サンプルの違い

- 棋譜は「対局の流れ」を保存するもの
- 学習サンプルは「局面ごとの教師信号」を保存するもの
- 1 つの棋譜から複数の学習サンプルが生成される
- 学習サンプルには、棋譜にはない mask や value ラベル候補などが追加される可能性がある

## 4. 棋譜データの考え方

### 棋譜の役割

棋譜は、1 対局の流れを再現可能な形で残すための基本データである。研究開発上は、単なる閲覧用ではなく、再実行、学習データ生成、デバッグ、テスト再利用の基礎データとして扱う。

### 最低限持ちたい情報

| 項目 | 内容 | 重要度 |
| --- | --- | --- |
| `game_id` | 対局を一意に識別する ID | 必須 |
| `format_version` | 棋譜データ形式の版 | 必須 |
| `engine_version` | 使用エンジンまたはルール実装の版 | 推奨 |
| `players` | 各プレイヤーの種別、名称、設定 | 必須 |
| `start_info` | 開始日時、初期条件、seed など | 推奨 |
| `end_info` | 終了日時、終了理由など | 推奨 |
| `result` | 勝敗結果、引き分け、勝者 | 必須 |
| `moves` | 手順列 | 必須 |
| `notes` | 備考、異常終了メモなど | 任意 |

### 手順列の考え方

手順列は少なくとも以下を扱えるようにしたい。

- 手数インデックス
- action 表現
- 手番プレイヤー
- 必要に応じて action 前後の簡易情報
- 必要に応じて局面ハッシュ

### 人間可読性

- 初期段階では、人が目で追って確認しやすいことを重視する。
- action の保存表現は、内部表現そのものではなく、人間可読な別表現を併記できると望ましい。
- 不具合調査時にログから状況を理解しやすい形式がよい。

### 再現性

- 同じ初期条件と手順列から同じ局面列を再構築できることが重要である。
- 棋譜だけで完全再現できるか、別途ルールバージョンや初期設定が必要かは明示する必要がある。
- 乱数要素が関与する場合は seed の保存を検討する。

### テスト再利用

- 棋譜は回帰テストの元データとして再利用しやすい。
- 過去に発生した不具合ケースの再現用資産にもなる。
- そのため、読み捨てではなく、再読込可能な保存形式を優先する。

## 5. 局面スナップショットの考え方

### 局面スナップショットの役割

局面スナップショットは、ある時点の state を単独で保存するためのデータである。1 局全体を保持する棋譜とは異なり、特定局面を素早く参照・共有・再現する用途に向く。

### 主な用途

- デバッグ
- 回帰テスト
- 学習データ生成の途中確認
- 既知局面の共有
- 不具合再現ケースの保存

### 最低限持ちたい情報

- `snapshot_id`
- `format_version`
- `engine_version`
- `state`
- 必要に応じて `source_game_id`
- 必要に応じて `ply_index` または `move_index`
- 必要に応じて `notes`

### 設計上の考え方

- `state` の保存形式は `engine-spec.md` の state シリアライズ方針と整合させる。
- 棋譜から再生成できる局面でも、頻出のテストケースはスナップショットとして独立保存できると便利である。
- キャッシュや一時的な派生情報は原則含めない。

## 6. 学習用サンプルの考え方

### 学習用サンプルの役割

学習用サンプルは、モデル学習のために局面単位で切り出したデータである。初期段階では、kobalab CPU の選択手を教師信号とする模倣学習が有力であり、そのための policy 学習データが中心になる想定である。

将来的には、自己対戦や強化学習に発展する可能性があるため、後から value 学習や結果ラベルも付与しやすい設計が望ましい。

### 最低限整理しておきたい項目

| 項目 | 内容 | 現時点での扱い |
| --- | --- | --- |
| `sample_id` | サンプル識別子 | 推奨 |
| `source_game_id` | 元対局 ID | 推奨 |
| `ply_index` | 対局中の何手目か | 推奨 |
| `state` または `observation_ref` | 入力元局面 | 必須 |
| `current_player` | 手番 | 必須 |
| `legal_actions` または `action_mask` | 合法手情報 | 重要 |
| `target_action` | 教師手 | 初期段階では必須に近い |
| `final_result` | 最終勝敗または結果ラベル | 将来用候補 |
| `value_target` | value 学習用ラベル | 候補、未確定 |
| `metadata` | 生成方法、教師元、設定 | 推奨 |

### 入力局面

- 入力は局面そのものでもよいし、局面から派生した観測データでもよい。
- 初期段階では、まず元の state を追跡できる設計が望ましい。
- 観測テンソルだけを保存して state を捨てる設計は、再現性やデバッグ性を下げる可能性がある。

### 手番

- 同じ盤面表現でも、手番が異なれば意味が変わる可能性があるため、明示的に保持する。
- state に含まれている場合でも、学習サンプル側で参照しやすいよう整理しておく価値がある。

### 合法手情報または action mask

- 学習時に不正手を防ぐため、legal action 情報は重要である。
- 保存方法としては以下が候補である。

| 候補 | 利点 | 懸念 |
| --- | --- | --- |
| legal action の列を保存 | 意味が明確 | 学習時に再符号化が必要 |
| action mask を保存 | 学習に使いやすい | action 空間定義が未確定だと固定しにくい |
| 必要時に state から再生成 | 保存量を減らせる | エンジン版差異の影響を受ける |

初期段階では、少なくとも legal action を再生成可能であることを前提としつつ、保存するかどうかは用途別に決めるのが現実的である。

### 教師手

- 模倣学習では、kobalab の選択手などを `target_action` として保存する。
- 人間可読表現、構造化表現、学習用 ID 表現のどこまで持つかは未確定である。
- 初期は、後から再解釈しやすいよう、可能なら構造化表現を優先する。

### value 相当情報

- 初期段階では policy 学習が中心だが、後から value 学習にも使える余地を持たせたい。
- 候補としては、最終勝敗、各プレイヤー視点の結果、スコア差などがある。
- ただし、最初からすべての value ラベルを固定する必要はない。

### 保存粒度

- 1 対局から複数サンプルを生成する。
- 保存単位としては、1 サンプル 1 レコードが扱いやすい。
- ただし、同一対局由来のサンプルをまとめて管理できるよう、`source_game_id` や分割単位の設計が必要である。

### policy 学習 / value 学習への将来拡張

- 初期は policy ターゲット中心でよい。
- ただし、後から value ターゲットを追加しやすい列構成を意識する。
- 学習用サンプル形式を「局面」「合法手情報」「教師信号」「由来メタデータ」に分けて考えると拡張しやすい。

## 7. 評価戦結果データの考え方

### 役割

評価戦結果データは、プレイヤー同士の比較結果を、個々の棋譜とは別に集計したものである。学習モデル vs kobalab、学習モデル vs random、学習モデル旧版 vs 新版などの比較結果を、後から追跡しやすくするために必要である。

### 想定する比較例

- モデルA vs モデルB
- 学習モデル vs kobalab
- 学習モデル vs random
- 学習モデル vs rule-based CPU

### 最低限持ちたい項目

| 項目 | 内容 |
| --- | --- |
| `evaluation_id` | 評価実行 ID |
| `executed_at` | 実行日時 |
| `format_version` | 評価結果形式の版 |
| `match_config` | 対戦条件、試合数、先手後手条件、seed 条件 |
| `player_a` | プレイヤー A の情報 |
| `player_b` | プレイヤー B の情報 |
| `game_count` | 実施対戦数 |
| `summary` | 勝敗集計、引き分け数 |
| `by_side` | 先手後手別結果 |
| `artifacts` | 参照する棋譜群やログ群 |
| `notes` | 備考 |

### 保存の考え方

- 個別対局の棋譜は別に保存し、その上に評価集計データを持つ構成が望ましい。
- summary だけではなく、どの棋譜群を元にしたか辿れることが重要である。
- モデル名だけでなく、モデルバージョン、チェックポイント、設定差分も記録したい。

## 8. 保存形式の候補

用途ごとに向く形式は異なるため、1 つに断定しすぎず整理する。

| 形式 | 向いている用途 | 利点 | 懸念 |
| --- | --- | --- | --- |
| JSON | 棋譜、局面スナップショット、設定、評価サマリ | 人間可読、扱いやすい | 大量データでは重い |
| JSON Lines | 学習サンプル、イベントログ | 追記しやすく分割しやすい | スキーマ管理が緩くなりやすい |
| CSV | 単純な集計結果、表形式メタデータ | 軽くて扱いやすい | ネスト構造に弱い |
| Parquet | 大量学習サンプル、分析用データ | 圧縮、列指向、高速集計に向く | 初期導入コストがある |

### 現時点での温度感

- 棋譜、局面スナップショット、評価サマリは JSON 系が第一候補
- 大量の学習サンプルは JSONL を第一候補としつつ、将来的に Parquet も検討余地あり
- 単純な評価集計表は CSV を補助的に使ってもよい

### 用途別の整理

- 人間が読む、差分を見る、テスト資産にする: JSON が扱いやすい
- 逐次生成する、大量に書き出す: JSONL が扱いやすい
- 学習向けに大量処理する: JSONL から Parquet への変換も候補

## 9. バージョン管理と互換性

### データ形式変更の前提

本プロジェクトでは、初期段階でデータ形式が変わる可能性が高い。したがって、少なくとも主要データ形式には `format_version` を持たせることが望ましい。

### version として持ちたいもの

- データ形式の版
- エンジンまたはルール実装の版
- 必要に応じて action encoding の版
- 必要に応じて observation 生成の版

### engine ルールバージョンとの関係

- 同じ棋譜や局面でも、エンジン側のルール解釈が変わると再現結果が変わる可能性がある。
- そのため、データ形式 version と engine version は分けて管理する方が望ましい。

### 後方互換の考え方

- 初期段階では完全な後方互換を優先しすぎない。
- ただし、過去データが読めなくなった理由を追跡できるようにする。
- 変換スクリプトを用意しやすい構造にしておくことが重要である。

## 10. テスト・再現性との関係

### テスト資産としてのデータ

データは学習だけでなく、テストと再現性のための資産でもある。特に以下の用途を意識する。

- 既知局面の保存
- 不具合再現用データの保持
- シリアライズ往復検証
- 同じ棋譜から同じ局面に再現できることの確認

### 既知局面の保存

- 小さなスナップショットとして保存し、回帰テストで使いやすくする。
- 期待される legal_actions や勝敗判定とセットで管理できると望ましい。

### 不具合再現用データ

- 例外発生局面、異常終了対局、合法手不整合ケースなどは保存価値が高い。
- 再現に必要な条件をメタデータとして残す。

### シリアライズ往復検証

- 保存した state や棋譜を読み戻して同じ内容になることを確認する。
- 特に action 表現と state 表現の往復は重点的に確認したい。

### 再現性の基本要件

- 同じ初期条件、同じ手順列、同じルールバージョンで同じ結果になること
- 乱数が関与する場合は seed と生成条件を追跡できること

## 11. 未確定事項

現時点で未確定または要検討の事項は以下の通りである。

- state の最終保存表現
- action の最終保存表現
- legal action を保存するか再生成前提にするか
- action mask を初期段階から保存するかどうか
- value ラベルを初期段階から持つかどうか
- 学習用データの最終ストレージ形式
- データ分割単位
- 圧縮戦略
- ファイル命名規則
- 大規模データ生成時のシャーディング方針

未確定事項は、実装中に暗黙で固定しない。`engine-spec.md` と突き合わせながら決める。
