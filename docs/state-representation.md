# MOSAIC state 表現整理メモ v0

## 1. 文書の目的

本書は、MOSAIC 学習型 CPU プロジェクトにおける通常 2人用 7×7 の Python エンジン state 表現を整理するための文書である。目的は、盤面保持、合法手列挙、自動配置判定、連鎖処理、終局判定、シリアライズ、学習接続、bridge 連携の基礎になる「エンジン内部 state」を実装前に明確にすることにある。

`docs/engine-spec.md` はエンジン責務全体を扱い、`docs/coordinate-system.md` は座標系を扱い、`docs/data-format.md` と `docs/sample-data.md` は保存形式と概念例を扱う。本書は、それらの前提を受けて、state を独立論点として具体化する役割を持つ。

state 表現を独立に整理する理由は、state が学習入力、保存データ、bridge 用交換表現とは目的が異なるためである。本書では、まず「正しく扱いやすいエンジン state」を優先し、学習用観測や外部交換表現とは同一視しない。

## 2. state が持つべき責務

通常 2人用 7×7 のエンジン state は、少なくとも以下を責務として持つ。

| 項目 | 役割 |
| --- | --- |
| 盤面占有状況 | 各位置に何が置かれているかを保持する |
| 手番情報 | 現在行動するプレイヤーを示す |
| プレイヤー情報 | 各プレイヤーの残り駒数などを保持する |
| 終局状態 | ゲーム終了済みかを示す |
| 勝者情報 | 勝敗が確定したか、誰が勝者かを示す |
| 履歴または参照情報 | 再現、デバッグ、ログ連携に必要な最小情報を持つ |
| 自動配置処理に必要な情報 | 最小正方形判定と連鎖処理が可能な状態を持つ |

### 重要な考え方

- state は、ルールに従って次状態を求めるための一次情報を持つ
- legal_actions や自動配置候補のような派生情報は、原則として state から計算できるようにする
- 学習用テンソルや bridge 用 JSON は state から変換する前提で考える

## 3. 盤面占有情報の持ち方の候補

盤面占有情報の持ち方には複数案がある。

### 候補A: occupied position を列挙する案

例のイメージ:

- `[{level: 0, x: 3, y: 3, owner: "neutral"}, ...]`

利点:

- JSON 化しやすい
- 差分確認しやすい
- bridge や保存形式に近い

欠点:

- 位置参照のたびに探索が必要になりやすい
- legal_actions や最小正方形判定で毎回変換が必要になる可能性がある

### 候補B: 各 level ごとの 2 次元配列で持つ案

例のイメージ:

- `board[level][y][x] = occupant`

利点:

- 位置参照が素直で速い
- `coordinate-system.md` の `(level, x, y)` と整合しやすい
- 2×2 最小正方形判定や支持条件判定が書きやすい

欠点:

- シリアライズ時に多少冗長になることがある
- 各 level サイズが異なるため、構造を揃える工夫が必要

### 候補C: 辞書的に持つ案

例のイメージ:

- `occupied[(level, x, y)] = occupant`
- または JSON 向けに `{"L0:3:3": "neutral"}` のような形式

利点:

- 占有位置だけを持てる
- 空き判定との相性は悪くない
- 内部では扱いやすい可能性がある

欠点:

- 順序や走査の扱いが少し見えにくい
- JSON ではタプルキーがそのまま使えない
- 2×2 判定では結局座標展開が必要

### 暫定方針

通常 2人用 7×7 では、**各 level ごとの 2 次元配列で持つ案**が最も現実的である。

理由:

- 盤面サイズが小さく、速度より可読性と実装しやすさを優先しやすい
- 最小正方形判定、支持条件判定、走査順が自然に書ける
- `level / x / y` 座標系と噛み合う

## 4. プレイヤー状態の持ち方

### 最低限持ちたい情報

| 項目 | 内容 |
| --- | --- |
| `player_id` | プレイヤー識別子 |
| `remaining_pieces` | 残り駒数 |
| `color` または `owner_label` | 表示や保存用の識別 |

### 中立コマの扱い

中立コマはプレイヤー状態ではなく、盤面占有者の 1 種として扱うのが自然である。

候補:

- `owner = 0 / 1 / neutral`
- `occupant_type = player / neutral`

### 暫定方針

- 各プレイヤー状態は `remaining_pieces` を明示的に持つ
- 中立コマは player とは別の occupant として盤面側で表現する
- `owner` は内部的に整数でもよいが、保存やログでは意味が追いやすい表現へ変換できるようにする

## 5. 手番とゲーム進行状態

### state に持ちたい項目

- `current_player`
- `turn_index` または `ply_index`
- `game_over`
- `winner`

### どこまで state に持つか

- `current_player` は必須
- `ply_index` は再現性とログのために持つ価値が高い
- `game_over` は派生でも求められるが、毎回計算するより state に持つ方が扱いやすい
- `winner` も同様に、終局時に確定した結果として保持してよい

### 派生計算で済ませるものとの切り分け

- 現局面での legal_actions
- 自動配置候補
- 最小正方形一覧

これらは原則として state から都度計算する想定とし、常時 state に保持しない方針がシンプルである。

## 6. 履歴の扱い

### 履歴を state に含めるか

履歴の扱いには複数の選択肢がある。

| 案 | 内容 | 利点 | 懸念 |
| --- | --- | --- | --- |
| 完全履歴を持つ | すべての action 履歴を state に含める | 再現しやすい | state が重くなる |
| 直近 action のみ持つ | 最後の action だけ保持する | デバッグに有用 | 完全再現には不足 |
| 履歴は外部ログに任せる | state は現局面中心 | シンプル | state 単体では再現しにくい |

### 暫定方針

- engine 内部の `GameState` は、現局面を中心に持つ
- 完全履歴は外部ログや棋譜データ側の責務とする方向が自然である
- ただし、`last_action` や `ply_index` のような最小参照情報を持つのは有用である

### 責務分離

- エンジン内部 state: 今の局面を正しく扱うための構造
- 棋譜データ: 対局全体の流れを保存する構造

この分離により、state を過度に肥大化させずに済む。

## 7. 派生情報の扱い

### 都度計算する候補

- legal_actions
- 最小正方形候補の成立判定
- 自動配置候補
- action mask の元になる legal set

### キャッシュするかどうか

初期段階では、正しさとシンプルさを優先し、派生情報は基本的に都度計算する方針が望ましい。

理由:

- 通常 2人用 7×7 は盤面規模が小さい
- キャッシュ整合性のバグを避けやすい
- テスト時に期待値を追いやすい

### キャッシュを検討する場面

- 大量自己対戦で速度が問題になったとき
- 同一 state に対して同じ派生情報を何度も使うとき

ただし、その段階でもキャッシュは state 本体と混ぜず、補助情報として扱う方が安全である。

## 8. シリアライズと保存の観点

### 基本方針

- JSON に落としやすいこと
- 再現性が高いこと
- テスト fixture 化しやすいこと

### data-format / sample-data とのつながり

- `docs/data-format.md` では state をスナップショットや学習サンプルの一部として扱う
- `docs/sample-data.md` では state の概念例を示している
- 本書は、それらに対して「内部ではどう持つのが扱いやすいか」を整理する

### format_version について

state の保存形式には、必要に応じて `format_version` を持たせる価値がある。特に、内部 state をそのまま保存用に出すのではなく、保存用スキーマへ変換する場合は version 管理しやすくなる。

### 保存しやすさと内部責務のバランス

- 保存しやすさだけを優先して内部表現を選ばない
- ただし、最終的に JSON へ素直に変換できる構造を意識する

## 9. 学習基盤との接続

### 学習用観測との関係

- state は学習用観測テンソルそのものではない
- 観測テンソルは state から変換して生成する
- この分離により、学習方針が変わってもエンジン state を保ちやすい

### action mask 生成との関係

- action mask は legal_actions から生成する
- そのため、state は legal_actions を正しく計算できるだけの情報を持っていればよい

### bridge 交換表現との違い

- bridge では JSON 互換の交換表現が必要になる
- ただし、その表現は内部 state の写像先であり、内部 state と同一である必要はない

### 実装上の姿勢

- エンジン内部 state をまず整える
- 学習用入力や bridge request はそこから変換する
- 内部 state を学習都合で歪めない

## 10. 現時点での暫定方針

通常 2人用 7×7 で最も扱いやすい暫定方針は以下である。

- 盤面は `level` ごとの 2 次元配列で持つ
- 位置の意味は `docs/coordinate-system.md` の `(level, x, y)` に揃える
- 各セルには `empty / player0 / player1 / neutral` のような occupant 情報を持てるようにする
- プレイヤー状態として `remaining_pieces` を持つ
- `current_player`、`ply_index`、`game_over`、`winner` を state に持つ
- legal_actions や自動配置候補は都度計算する
- 完全履歴は外部ログへ寄せ、state には最小参照情報だけ持たせる

この方針は、正しく扱いやすいエンジン state を優先しつつ、保存や bridge への変換もしやすい。
