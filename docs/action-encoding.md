# MOSAIC action 表現整理メモ v0

## 1. 文書の目的

本書は、MOSAIC 学習型 CPU 研究開発プロジェクトにおける action 表現の整理を目的とした初期仕様メモである。action は、Python エンジン内部の状態遷移、合法手列挙、学習用 action mask、棋譜保存、教師手記録、kobalab との入出力、ログ表示にまたがる中心論点であり、1 箇所でまとめて整理しておく必要がある。

`docs/engine-spec.md` はエンジン全体の責務整理、`docs/data-format.md` は棋譜や学習サンプルの保存整理、`docs/kobalab-bridge.md` は Python と kobalab の橋渡し整理を扱う。本書は、それらで共有される action 論点を独立文書として切り出し、各文書の前提を揃える役割を持つ。

ルールの一次資料は [game-rules-standard.md](/d:/projects/mosaic_ai/docs/game-rules-standard.md)、座標系の一次資料は [coordinate-system.md](/d:/projects/mosaic_ai/docs/coordinate-system.md) とする。本書は、それらを参照しながら action の内部表現、保存 / 交換用表現、学習用表現を整理する位置づけとし、ルール詳細や座標詳細を本書側に抱え込みすぎない。

現時点では実装前の設計メモであり、最終フィールド定義や整数 ID の割当は未確定である。ただし、実装時に迷いにくいよう、表現レイヤーと比較観点は先に整理する。

当面の対象は通常 2人用・7×7 の MOSAIC のみとし、mini、quo、多人数差分は本書の主対象外とする。

## 2. action が関係する場面

action は少なくとも以下の場面に関わる。

| 場面 | action に求められること |
| --- | --- |
| Python エンジン内部 | state に適用できること、検証しやすいこと |
| legal actions の列挙 | 列挙結果と apply_action が整合すること |
| 学習用 action mask | 固定長表現または対応関係へ落とし込みやすいこと |
| 棋譜保存 | 手順列として追跡しやすく、再現に使えること |
| 学習サンプルの教師ラベル | 教師手として記録しやすく、後から再解釈できること |
| kobalab との入出力 | JSON などで受け渡ししやすく、意味が明確であること |
| ログ・デバッグ表示 | 人間が読んで状況を追いやすいこと |

### この文脈で重要な点

- 1 つの表現ですべてを無理に満たそうとすると扱いにくくなる可能性が高い
- 一方で、表現を増やしすぎると変換が複雑になる
- そのため、役割ごとにレイヤーを分けつつ、変換関係を明示する方針が有力である
- 通常 2人用 7×7 では、手番で選ぶ action と着手後の自動配置処理を分けて扱うことが重要である

## 3. action の基本要件

action 表現には少なくとも以下の要件がある。

- 着手内容を一意に表せること
- エンジンで適用しやすいこと
- legal_actions の列挙結果と整合すること
- 人間が読んで追いやすいこと
- 学習時に整数化しやすいこと
- 棋譜保存や bridge で扱いやすいこと
- 将来の特殊ケースや拡張に耐えやすいこと

通常 2人用 7×7 を前提にすると、手番で選ぶ action は少なくとも以下の通常配置候補を一意に表せる必要がある。

- 最下層の空き位置
- 直下 4 点支持が成立した上層の空き位置

一方で、自動配置はプレイヤーが直接選ぶ action ではなく、着手後処理として扱う前提を維持する。

### 要件間のトレードオフ

| 要件 | 重視される場面 | 注意点 |
| --- | --- | --- |
| 一意性 | エンジン、棋譜、再現 | 曖昧な省略表現は避けたい |
| 構造性 | エンジン、検証、bridge | 学習では別変換が必要になりやすい |
| 可読性 | ログ、棋譜、デバッグ | 内部表現そのままでは読みにくいことがある |
| 固定長性 | 学習、mask 生成 | ルール未確定だと早期固定が難しい |
| 拡張性 | 長期保守 | 初期段階で複雑化しすぎないことも重要 |

## 4. 表現レイヤーの分離案

現時点では、action を 1 つの表現に統一するより、少なくとも以下の 3 レイヤーに分けて考える案が現実的である。

| レイヤー | 主用途 | 想定する性質 |
| --- | --- | --- |
| 内部表現 | エンジン内の apply_action、legal_actions、検証 | 構造化され、意味が明確 |
| 保存 / 交換用表現 | 棋譜保存、JSON ログ、bridge 入出力 | JSON 化しやすく、人間可読 |
| 学習用エンコード表現 | action mask、モデル出力、教師ラベル | 整数 ID または fixed-length index |

### 分離する理由

- エンジン内部では意味を失わない構造化表現が必要
- 保存や bridge では JSON 化しやすく追跡しやすい表現が必要
- 学習では固定長に落とし込める形式が必要
- 座標は [coordinate-system.md](/d:/projects/mosaic_ai/docs/coordinate-system.md) の `level / x / y` 前提に整合させる必要がある

### 想定する変換関係

- 内部表現 -> 保存 / 交換用表現
- 保存 / 交換用表現 -> 内部表現
- 内部表現 -> 学習用エンコード表現
- 学習用エンコード表現 -> 内部表現

このとき、保存 / 交換用表現と学習用表現を直接結びすぎず、可能なら内部表現を基準に考える方が整合を取りやすい。

## 5. 内部表現の候補

### 役割

内部表現は、Python エンジン内で `legal_actions` が返し、`apply_action` が受け取り、検証ロジックが解釈する中心的な表現である。意味が曖昧でないことが最優先になる。

### 候補

| 候補 | 特徴 | 利点 | 懸念 |
| --- | --- | --- | --- |
| 構造化オブジェクト | `dataclass` 的な明示的フィールドを持つ | 意味が明確、型で整理しやすい | 初期にフィールド設計が必要 |
| タプル的表現 | 順序ベースの軽量表現 | 実装が軽い | 可読性が低く、将来拡張に弱い |
| 辞書的表現 | JSON に近い柔軟な表現 | 柔軟で bridge と近い | 型の厳密性が下がる |

### 現時点での見方

- 内部表現の第一候補は、構造化オブジェクトまたは構造化辞書である
- 単純なタプルだけで押し通すのは、可読性と拡張性の面でやや不利である
- ルール細部が未確定な段階では、最初からフィールドを過度に固定しすぎないことも重要である

### 内部表現で持ちたい性質

- action 種別が分かること
- 主要パラメータが明示されること
- デバッグ時に `repr` 相当で追いやすいこと
- バリデーションしやすいこと
- 位置を持つ場合、`level / x / y` を自然に含められること

### 柔軟性について

MOSAIC の 1 手が単一操作で表せるか、複合的選択を含むかは未確定である。そのため、内部表現では少なくとも以下のどちらかに耐えられる必要がある。

- 単一 action 種別 + パラメータ群
- action 種別 + 複数フィールド、またはサブアクション列

通常 2人用 7×7 の当面の実装では、プレイヤーが選ぶ action は通常配置 1 手とし、その位置は `level / x / y` で表せる前提が最も自然である。

## 6. 保存・交換用表現の候補

### 役割

保存・交換用表現は、棋譜、学習データ、bridge 入出力、ログで扱うための表現である。内部表現よりも、人間可読性、JSON 化のしやすさ、バージョン差異への強さが求められる。

### 候補

| 候補 | 利点 | 懸念 |
| --- | --- | --- |
| 明示的な JSON オブジェクト | フィールド名が見え、追跡しやすい | 冗長になりやすい |
| 短い文字列表現 | 棋譜やログで見やすい | 意味の厳密性が落ちやすい |
| JSON + 可読文字列併記 | 厳密性と可読性を両立しやすい | 保存量は増える |

### 保存・交換用表現で重視すること

- JSON 化しやすいこと
- ログ追跡しやすいこと
- `format_version` や action encoding version を持たせやすいこと
- bridge の異常系でも内容を確認しやすいこと
- 位置を [coordinate-system.md](/d:/projects/mosaic_ai/docs/coordinate-system.md) と整合する形で表せること

### 棋譜向けの温度感

- 棋譜保存では、人間が流れを追いやすいことが重要である
- そのため、構造化 JSON を主としつつ、必要なら可読文字列を補助で持つ構成が相性よい

### bridge 向けの温度感

- bridge では、機械が厳密に解釈できることが優先である
- したがって、短縮文字列だけに依存せず、フィールド名を持つ JSON 互換表現が望ましい

## 7. 学習用表現の候補

### 学習用表現が必要な理由

モデル出力や action mask では、action を整数 ID や fixed-length な index に落とし込む必要が出てくる。内部表現や JSON 表現のままでは、ニューラルネット入出力として扱いにくい。

### 候補方針

| 方針 | 内容 | 利点 | 懸念 |
| --- | --- | --- | --- |
| 全 action 空間を固定長で持つ | すべての可能 action に ID を割る | mask と相性が良い | 空間設計が難しい |
| 局面依存候補集合を別管理する | legal actions の並びに対する index を使う | 初期実装が軽い | モデル出力設計が複雑になる |
| ハイブリッド | 大枠は固定長、細部は追加情報で表現 | 柔軟性がある | 実装複雑度が上がる |

### 現時点での考え方

- 最終決定はまだしない
- ただし、将来の action mask を考えると、固定長 action 空間の可能性は早めに検討しておく必要がある
- 一方で、ルール未確定の段階で無理に固定長を断定するのは避けたい

通常 2人用 7×7 では、固定長化を検討する際も、まずは `level / x / y` の座標系と legal_actions の列挙結果を基準に action 空間を考えるのが自然である。

### 学習用整数 ID に備えるための要件

- 内部表現から一意に encode できること
- decode して元の意味に戻せること
- legal_actions と mask の対応が曖昧にならないこと
- version 管理しやすいこと

## 8. legal actions / action mask との関係

### 整合性の原則

- `legal_actions` が返す action は、内部表現としてそのまま `apply_action` に渡せることが望ましい
- 学習用 action mask は、`legal_actions` から一意に生成できることが望ましい
- この対応関係が曖昧になると、学習と実行がずれやすい

通常 2人用 7×7 では、`legal_actions` が返すのは通常配置として選択可能な action であり、自動配置はここに含めない前提とする。

### mask の生成元

mask の生成元としては以下が考えられる。

- `legal_actions` を内部表現のまま受け取り、別途 encode して mask を作る
- `legal_actions` 取得時にあわせて encoded index も得る

現時点では、エンジン本体よりも学習アダプタ側に寄せる余地があるが、少なくとも `legal_actions` と encoded action の対応表を安定して作れる必要がある。

### mask の保存要否

- 学習サンプルに mask を保存するかどうかは未確定
- 初期段階では、state から再生成可能であることを前提にしつつ、必要に応じて保存する方針が現実的である
- ただし、action encoding version が変わると再生成結果も変わる可能性があるため、version 情報は重要である

### 不正 action の検出

- decode した action が形式的に不正
- encode はできても当該局面では legal ではない
- bridge や棋譜読込で未知形式が来る

こうしたケースを区別できるようにしておく必要がある。

### 最優先事項

合法手列挙の正しさが最優先である。action encoding は、その正しい legal_actions を壊さずに学習へつなぐための補助層として扱う。

特に通常 2人用 7×7 では、最下層空き位置と直下 4 点支持がある上層空き位置を正しく action 候補にできることが重要である。

## 9. 棋譜・学習データ・bridge への影響

### 棋譜への影響

- 棋譜では、対局の流れを再現できる粒度で action を保存したい
- 人間が確認しやすい表現も欲しい
- したがって、保存・交換用表現をそのまま使う方針が自然である

### 学習データへの影響

- 学習サンプルでは、教師手を後から再解釈できることが重要である
- そのため、整数 ID だけでなく、必要なら元 action への参照や version 情報も持つ方が安全である
- 初期段階では、`target_action` を構造化表現で持ちつつ、別途 encoded label を生成する流れが有力である

### bridge への影響

- bridge では、Python と kobalab の間で厳密な意味を共有する必要がある
- JSON 互換の明示的表現を前提にした方が誤解が少ない
- 文字列表現は補助的に使うに留めるのが無難である
- 位置情報は、可能なら `level / x / y` を持つ交換表現へ寄せる方が整合を取りやすい

### 同一表現にするか、変換前提にするか

すべてを同一表現に揃えるより、以下の整理が現実的である。

- エンジン内部: 構造化された action
- 棋譜 / bridge: JSON 向きの保存・交換用表現
- 学習: 別途 integer encoding

この場合、変換を前提にする代わりに、変換の基準点を明示しておく必要がある。

## 10. 現時点での暫定方針

現時点で最も現実的な整理は、action を 3 レイヤーで扱う方針である。

### 暫定方針

- エンジン内部では、意味を持つ構造化された action を扱う
- 保存 / 交換用には、JSON 化しやすい明示的表現を使う
- 学習時には、別途 integer encoding を定義する
- 位置を持つ action は、[coordinate-system.md](/d:/projects/mosaic_ai/docs/coordinate-system.md) の `level / x / y` に整合する形で持つ
- 通常 2人用 7×7 では、プレイヤーが選ぶ action と自動配置処理を分けて扱う

### この方針の理由

- `engine-spec.md` の apply_action と legal_actions の整合を取りやすい
- `data-format.md` の棋譜、学習サンプル、ログ保存と相性が良い
- `kobalab-bridge.md` の JSON ブリッジ方針と噛み合う
- 学習都合だけに寄りすぎず、デバッグ性と可読性を保ちやすい

### まだ断定しない点

- 内部表現を `dataclass` にするか構造化辞書にするか
- 学習用整数 ID を全 action 固定長にするか
- action の最終フィールド定義

## 11. 未確定事項

現時点で未確定または要確認の事項は以下の通りである。

- action の最終フィールド定義
- MOSAIC における 1 action の自然な粒度
- 全 action 空間の固定方法
- 学習用整数 ID の具体的割当方式
- 局面依存 action と固定長 action 空間の整合方法
- bridge で必要な最小 action 項目
- 人間可読文字列表現をどこまで標準化するか
- 将来の特殊ケースや複合 action があるかどうか
- `level / x / y` をどこまで action の必須位置項目にするか

## 12. 次アクション

### 他文書と突き合わせて決めるべきこと

- `docs/engine-spec.md` と: `legal_actions` と `apply_action` が扱う内部 action の最小要件
- `docs/data-format.md` と: 棋譜保存と学習サンプル保存に必要な action 項目
- `docs/kobalab-bridge.md` と: bridge 入出力に必要な action 項目と粒度

### 実装前に最小限確定したいこと

- action の暫定内部表現
- 保存 / 交換用表現の最小 JSON スキーマ
- encode / decode の責務境界
- legal_actions と action mask の対応付け方針

### 切り出し候補文書

- `docs/action-examples.md`
  action の具体例とケース別整理
- `docs/action-mask.md`
  学習用 mask 生成方針の詳細
- `docs/action-schema.md`
  保存 / 交換用 JSON スキーマ案

本書は初版であり、最終仕様を固定するものではない。ただし、action を「内部表現」「保存 / 交換用表現」「学習用表現」に分けて考えること、legal_actions と action mask の整合を最優先で扱うこと、学習都合だけでなく可読性とデバッグ性も重視することは、当面の基本方針として維持する。
