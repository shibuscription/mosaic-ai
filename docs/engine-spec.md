# MOSAIC Python エンジン仕様書 v0

## 1. 文書の目的

本書は、MOSAIC 学習型 CPU 研究開発プロジェクトにおける Python 版エンジンの初期仕様を整理するための文書である。目的は、高速化や最適化を急ぐことではなく、今後の実装、テスト、自動対戦、データ生成、学習基盤整備の土台になる「正しく・扱いやすく・テストしやすい」エンジン像を明確にすることにある。

本書はルートの `README.md` を詳細化する位置づけであり、README が全体方針とロードマップの入口、本書が Python エンジンに関する詳細仕様の初版という役割分担を持つ。

ルールそのものの一次資料は [game-rules-standard.md](/d:/projects/mosaic_ai/docs/game-rules-standard.md) とし、本書はそれを参照してエンジン責務、state、action、API、テスト観点を整理する位置づけとする。ルール詳細を本書側へ重複記載しすぎず、役割分担を保つ。

現時点では実装前の仕様整理文書であり、未確定事項は未確定のまま残す。ルール詳細が曖昧な箇所を無理に埋めることより、実装時に必要となる責務分解とインターフェース方針を先に揃えることを優先する。

当面の実装対象は、通常 2人用・7×7 の MOSAIC のみとする。mini、quo、多人数差分は本書の主対象外とし、必要になった時点で別途整理する。

## 2. Python エンジンの責務

### エンジンが持つ責務

- 盤面状態の保持
- 各プレイヤーに関する状態の保持
- 手番管理
- 合法手の列挙
- 着手適用による状態遷移
- 最小正方形判定
- 自動配置判定
- 連鎖処理
- 終局判定
- 勝敗判定
- 状態および履歴のシリアライズ
- テストしやすい API の提供
- 学習基盤、自動対戦、評価スクリプトから利用しやすい構造の提供

### この段階では責務に含めないもの

- 学習ロジックそのもの
- ニューラルネット推論
- MCTS や探索アルゴリズム本体
- GUI や可視化 UI
- kobalab CPU ロジック本体
- 評価リーグ全体の運営ロジック
- データセット変換や学習向け前処理の最終仕様

### 責務の考え方

本エンジンは、MOSAIC を「正しく進行させること」に責務を集中させる。プレイヤーの強さや学習方式はエンジンの外側で扱う。そのため、エンジンは特定の CPU 実装に依存せず、任意のプレイヤー実装が同じ状態と action を扱えることを目指す。

## 3. 状態管理の考え方

### state が持つべき主要情報

現時点では、`GameState` に相当する単位が少なくとも以下を保持する想定とする。

| 項目 | 内容 | 備考 |
| --- | --- | --- |
| 盤面情報 | 盤上の配置、占有状況、ゲーム進行に必要な配置情報 | 具体表現は未確定 |
| プレイヤー状態 | 各プレイヤーに属する状態、残りコマ数、必要に応じて補助情報など | 通常 2人用 7×7 を基準に整理する |
| 手番情報 | 現在行動するプレイヤー | 必須 |
| 手数 | 現局面までの action 数 | 再現性とログ用に有用 |
| 履歴 | 過去の action、必要に応じて局面差分または局面スナップショット | 粒度は未確定 |
| 終局状態 | 終局済みかどうか | 必須 |
| 勝者情報 | 勝者、引き分け、未確定など | 表現は要設計 |
| 補助情報 | キャッシュ、派生値、検証用メタ情報 | 永続化対象にするかは分離推奨 |

通常 2人用 7×7 を対象とする当面の実装では、初期中立コマ、各層の占有状態、現在手番、残りコマ数、自動配置連鎖後に確定する終局情報を素直に保持できることが重要である。

### 状態設計の基本方針

- 状態は、可能な限りエンジン単体で完結して検証できるようにする。
- 学習用特徴量は state の外部で生成可能にしつつ、必要な生情報は state から取得できるようにする。
- 派生情報は、原則として一次情報から再計算できるものと、保持しないと扱いにくいものを分けて考える。
- キャッシュや一時的な計算結果は、シリアライズ対象の本体状態と混ぜない方針が望ましい。

### 盤面内部表現の考慮点

盤面内部表現は未確定だが、少なくとも以下を比較対象とする。

- Python の `dataclass` とリストやタプルを用いた素直な構造
- 不変性を意識したタプル主体の構造
- NumPy 配列を前提とした構造
- シリアライズしやすい辞書主体の構造

初期段階では、学習向けテンソル最適化よりも、可読性、比較しやすさ、テストしやすさを優先する。

### 履歴の扱い

履歴は少なくともデバッグ、再現、棋譜保存に必要である。一方で、state 本体にどこまで履歴を含めるかは未確定である。

候補:

- action のみを順次保持する
- action と主要イベントだけを保持する
- 局面スナップショットも一定間隔または毎手保持する

初期実装では、まず action 履歴を保持し、局面スナップショットは外部ログまたは別用途の仕組みで扱う案が現実的である。

## 4. action 表現の考え方

### 基本方針

action は、エンジン内部で正しく適用できることに加え、学習用に整数化しやすく、人間がログを読んだときにも追いやすいことが望ましい。そのため、内部表現と外部表現を分ける前提で設計する余地を残す。

### 検討すべき観点

- action を 1 手単位で表すか、複合的な選択をまとめて 1 action にするか
- 複数段階の意思決定がある場合、1 action に畳むか、中間状態を導入するか
- 学習時に固定長の action 空間へ落とし込みやすいか
- デバッグ時に文字列化しやすいか
- 保存時に JSON 化しやすいか

### 表現案

| 観点 | 案 | 利点 | 懸念 |
| --- | --- | --- | --- |
| 内部表現 | 構造化オブジェクトまたは辞書 | 意味が明確で検証しやすい | 学習時には別途符号化が必要 |
| 外部表現 | 文字列または JSON 互換オブジェクト | ログ・保存・比較に向く | 厳密性は設計次第 |
| 学習表現 | 整数 ID または固定長インデックス | action mask と相性が良い | ルール変更に弱い可能性 |

### 推奨する整理

- エンジン内部では意味を持つ構造化 action を採用する。
- 学習や推論では、別途 `encode_action` / `decode_action` に相当する変換層を設ける。
- ログや棋譜保存では、人間可読な表現を別に持てるようにする。

### 未確定事項

- MOSAIC における 1 action の自然な単位
- action 空間を固定長にできるかどうか
- 中間選択を state に含める必要があるかどうか

## 5. 基本 API 案

具体的な関数シグネチャは未固定だが、少なくとも以下に相当する責務を持つ API 群を想定する。

| API の役割 | 想定責務 | 補足 |
| --- | --- | --- |
| 初期化 | エンジン設定やルール設定を受け取る | ルール差分があるなら設定化を検討 |
| 初期局面生成 | 新規ゲーム開始状態を作る | `initial_state()` 相当 |
| 合法手取得 | 現局面の legal actions を返す | 順序の安定性も要検討 |
| 着手適用 | action を適用し次状態を得る | 破壊的更新か非破壊かは要検討 |
| 終局判定 | 現局面が終了しているかを返す | `is_terminal()` 相当 |
| 勝者取得 | 勝者または結果を返す | 終局前の扱いを定義する |
| 状態複製 | state を独立コピーする | 探索やシミュレーションに必要 |
| 状態保存 / 読込 | state や棋譜を保存・復元する | JSON 互換を意識 |
| 文字列表現 | ログ・デバッグ向け整形出力 | 比較しやすさ重視 |

### API 設計の基本方針

- 入出力はできるだけ明示的にする。
- 破壊的変更を許す API と、非破壊で次状態を返す API を混同しない。
- テスト時に差分確認しやすいよう、安定した文字列表現や辞書表現を持てるようにする。
- 例外を投げる操作と、結果オブジェクトで失敗を返す操作の境界を整理する。

### `step` と `apply_action` の関係

初期設計では、以下のように役割を分けると扱いやすい。

- `apply_action`: 通常配置 1 手を受け取り、その結果として必要な自動配置判定と連鎖処理まで含めた状態遷移処理
- `step`: 学習環境や対戦ループ向けに、状態遷移結果、終局情報、追加メタ情報をまとめて返す高水準 API

この分離により、エンジン本体の責務を保ちながら、将来的な学習環境化にも接続しやすくなる。

### `clone` の位置づけ

- 探索、ロールアウト、検証、分岐テストのために state 複製は必須と考える。
- 複製後に元 state と独立していることが保証される必要がある。
- 浅いコピーで十分か深いコピーが必要かは内部表現に依存するため、実装時に注意する。

## 6. 合法手生成と検証

### legal_actions の役割

`legal_actions` は、現在の state から選択可能な action を完全かつ矛盾なく列挙する責務を持つ。学習、推論、探索、自動対戦、評価のすべてがここに依存するため、エンジン実装の中でも最重要要素の 1 つとして扱う。

通常 2人用 7×7 では、少なくとも以下の整理に整合する必要がある。

- 最下層では、空いている位置が通常配置候補になる
- 上層では、直下 4 点支持が成立し、かつ空いている位置が通常配置候補になる
- 自動配置は legal action ではなく、通常配置後に発生する着手後処理として扱う

最小正方形、自動配置条件、中立コマを含む場合の扱い、連鎖の基本ルールは [game-rules-standard.md](/d:/projects/mosaic_ai/docs/game-rules-standard.md) を参照し、本書ではそれをエンジン責務としてどう扱うかに集中する。

### action 検証の必要性

合法手列挙とは別に、単独 action に対する検証機能が必要である。

想定する役割:

- 受け取った action が形式的に妥当か確認する
- 現在の state に対して実行可能か確認する
- 不正理由を区別できるようにする

通常配置として受け付ける action と、着手後の自動配置処理を明確に分けることが重要である。

### 不正手の扱い

不正手への対応は用途別に整理する必要がある。

- エンジン内部テストでは、明確な例外または失敗結果を返す
- 自動対戦では、プレイヤー実装の不具合検出に使える形にする
- 学習時には、原則として legal mask で不正手選択を防ぐ前提にする

不正手を暗黙に補正する設計は避ける。

### 学習時の合法手マスクとのつながり

- `legal_actions` から、学習用 action 空間に対応した mask を生成しやすいことが重要である。
- そのため、legal action 列挙と action 符号化の対応関係は後で曖昧にならないようにする必要がある。
- 初期段階では、mask 生成の責務をエンジン本体に含めるか、学習用アダプタ層に置くかは未確定だが、少なくとも変換元となる情報はエンジンが安定して提供する必要がある。

### 最重要ポイント

合法手列挙の正しさは、性能改善より優先する。不正確な legal_actions は、学習データ、評価戦、自己対戦のすべてを汚染するため、最初に厳密に扱う。

加えて、着手後処理としての最小正方形判定、自動配置判定、連鎖処理も同じ重要度で扱う必要がある。通常 2人用 7×7 のルールでは、通常配置 1 手の後に自動処理が続くため、legal_actions の正しさだけではエンジンの正しさは担保できない。

## 7. 終局判定・勝敗判定

### エンジンが持つ責務

- 終局条件を満たしたかどうかの判定
- 終局時の結果確定
- 勝者、敗者、引き分けなどの結果表現
- 勝敗判定に必要な状態情報の保持または参照

通常 2人用 7×7 では、勝利条件は「先に自分のコマをすべて置き切ること」を前提とし、終局判定と勝敗判定は着手後の自動配置連鎖まで解決した時点で確定する方針が自然である。

### 評価スクリプト側に持たせない責務

- 勝敗ルールそのものの判定
- 終局条件の本体ロジック
- 曖昧な状態からの便宜的な結果補完

評価スクリプト側は、エンジンが返した結果を集計・比較する立場に留め、ルール判断を持ち込まない方針が望ましい。

### 勝敗判定に必要な状態情報

具体項目はルール要確認だが、少なくとも以下を見落とさないようにする。

- 各プレイヤーの得点または評価対象値
- 盤面上の最終状態
- 行動不能や手番終了に関する情報
- 引き分け条件に関わる情報

### 未確定ルールの扱い

- ルール詳細が未確定な箇所は、仕様書上で「要確認」と明記する。
- 実装時は `TODO` や暫定分岐で埋めるのではなく、先に文書を更新してから反映する。
- 終局条件や勝敗条件が複数解釈ありうる場合は、別紙化候補として扱う。

## 8. シリアライズ方針

### 基本方針

シリアライズは、学習データ生成、棋譜保存、デバッグ、再現テストのために重要である。初期段階では JSON などの人間が確認しやすい形式を優先し、読みやすさと再現性を重視する。

### 保存しやすい構造の考え方

- state は JSON 互換の辞書へ変換しやすい構造を意識する
- action も JSON 化しやすい構造または文字列表現を持てるようにする
- バージョン情報を含められる形にする
- 必要なら自動配置を含む着手結果や連鎖結果も追跡できるようにする

### 保存単位の候補

| 保存単位 | 用途 | 備考 |
| --- | --- | --- |
| 単一 state | デバッグ、回帰テスト、局面再現 | 最小単位 |
| 対局全体 | 棋譜保存、自己対戦ログ、学習データ生成 | 主用途になる可能性が高い |
| 局面列 | 局面単位学習、教師データ生成 | 加工済みデータとして利用しやすい |

### action 履歴の保存方針

- 少なくとも action の順序が再現できること
- 必要なら各 action 前後の state を再構築できること
- 対局ログと学習用サンプル生成で共用しやすいこと

### バージョン管理

将来的な仕様変更に備え、保存形式には `format_version` に相当する情報を持たせることが望ましい。初期段階では過剰な複雑化は避けつつも、後で後方互換性の問題を追跡しやすくする。

## 9. テスト観点

エンジンは最初からテスト前提で設計する。特に以下の観点を重視する。

| テスト観点 | 確認したい内容 |
| --- | --- |
| 初期局面の正しさ | 初期 state が仕様通りに生成されるか |
| 合法手列挙 | legal_actions が漏れや重複なく正しいか |
| 着手適用 | action 適用後の state 遷移が正しいか |
| 自動配置判定 | 最小正方形条件から正しい自動配置が発生するか |
| 連鎖処理 | 自動配置の連鎖が正しく解決されるか |
| 再現性 | 同一初期条件、同一 action 列で同じ結果になるか |
| 終局判定 | 終了条件を正しく検出できるか |
| 勝敗判定 | 勝者や引き分け判定が正しいか |
| シリアライズ往復 | 保存・読込後に同一 state を再現できるか |
| 既知局面回帰 | 既知局面で過去に正しかった挙動が崩れていないか |

### 特に重要なテスト

- 合法手列挙と action 適用の整合性
- 最小正方形判定と自動配置判定の整合性
- 連鎖処理の deterministic な再現性
- 複製した state の独立性
- 不正手に対する失敗挙動
- 終局局面で追加 action を許さないこと

### 既知局面テスト

ルール理解が進んだ段階で、既知局面と期待結果をセットにした回帰テストを蓄積する。これは将来、kobalab 比較やデータ生成の検証にも役立つ。

通常 2人用 7×7 では、特に以下を既知局面テストの重要候補とする。

- 上層配置の支持条件確認
- 中立コマを含む最小正方形判定
- 自動配置の複数候補が同時成立する局面
- 連鎖処理が発生する局面
- 同一局面・同一着手で常に同じ解決順になること

## 10. 学習基盤との接続を見据えた要件

### 接続しやすさの要件

- state から観測データへ変換しやすいこと
- legal_actions から action mask を作りやすいこと
- 自動対戦を大量に回しやすいこと
- random、kobalab、学習済みモデルなどのプレイヤー差し替えがしやすいこと
- step 単位のログ取得がしやすいこと

### 設計上の優先順位

- 速度よりもまず正しさ
- 最適化よりも責務分離
- 学習専用設計よりも汎用的な状態遷移 API
- CPU 実装への依存よりもプレイヤー差し替え可能性
- ルール一次資料との整合と deterministic な再現性

### 学習環境との接続イメージ

将来的には、エンジンをそのまま学習環境として使うか、エンジンを包む薄い環境アダプタを用意する想定である。現時点では後者の方が柔軟性が高い。

想定される分離:

- エンジン本体: state、legal_actions、apply_action、terminal 判定
- 学習アダプタ: observation 生成、action mask 生成、reward 付与、step 形式の整形

この分離により、模倣学習、自己対戦、強化学習のいずれにも接続しやすくなる。
