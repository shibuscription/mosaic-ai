# MOSAIC kobalab CPU 接続仕様メモ v0

## 1. 文書の目的

本書は、MOSAIC 学習型 CPU 研究開発プロジェクトにおいて、既存の kobalab CPU をどのように活用・接続するかを比較検討するための設計メモである。目的は、実装に先立って接続方式の候補、比較観点、受け渡しデータ、テスト観点を整理し、後続の実装判断をしやすくすることにある。

ルートの `README.md` は全体方針とロードマップの入口、`docs/engine-spec.md` は Python エンジンの責務整理、`docs/data-format.md` はデータ形式整理を担う。本書は、それらの間にある「既存 CPU をどう橋渡しするか」という実務上の論点を扱う。

現時点では接続方式比較のための文書であり、最終方式を断定するものではない。未確定事項は未確定のまま残しつつ、比較軸と暫定方針を先に揃えることを重視する。

## 2. kobalab CPU の位置づけ

本プロジェクトにおける kobalab CPU の位置づけは、主に以下の 4 つである。

| 位置づけ | 役割 | 主な利用フェーズ |
| --- | --- | --- |
| 教師役 | 模倣学習向けの教師手生成 | Phase 2-3 |
| 対戦相手 | 学習モデルや他プレイヤーとの対戦 | Phase 2 以降 |
| ベンチマーク相手 | 性能比較の基準 | Phase 4 以降 |
| データ生成器の一部 | kobalab vs kobalab などで棋譜や局面を生成 | Phase 2 以降 |

### 温度感

- kobalab CPU は、本プロジェクトの中核実装そのものではない。
- あくまで既存の強い外部資産として尊重し、教師役、比較対象、データ生成源として活用する。
- 初期段階では、kobalab の意思決定を正しく再利用できることの方が、実装の美しさや速度最適化より重要である。
- Python エンジン側は独立して正しく実装し、その上で kobalab と接続する形を基本とする。

## 3. 接続方式の候補

### 候補A: Python にロジックを移植してネイティブ利用する

#### 概要

kobalab CPU のロジックを Python 側へ移植し、Python エンジンやプレイヤー実装から直接呼び出す方式である。

#### 利点

- Python エンジンとの統合度が高い
- subprocess や IPC が不要になり、運用が単純になる
- デバッグやプロファイルを Python 側で一元化しやすい
- 将来的な最適化や学習環境との密結合には向く可能性がある

#### 欠点

- 既存ロジックとの一致性を保ちながら移植するコストが高い
- 移植バグにより「kobalab を使っているつもりで別物になる」リスクがある
- テスト資産が不十分だと検証負荷が重い
- 初期段階での着手コストが高い

#### 評価

| 観点 | 評価 |
| --- | --- |
| 初期導入コスト | 高い |
| 長期保守性 | 条件次第で高い |
| データ生成との相性 | 良い可能性がある |
| テストしやすさ | 移植後は高いが、移植検証が難しい |

### 候補B: JavaScript / Node のまま保持し、CLI もしくは subprocess 的に呼び出す

#### 概要

kobalab CPU を既存実装のまま保持し、Python 側から Node プロセスを都度起動して問い合わせる方式である。

#### 利点

- 既存ロジックをそのまま使いやすい
- 移植コストが不要
- 初期の疎通確認を最小構成で始めやすい

#### 欠点

- プロセス起動コストが大きい可能性がある
- 呼び出し単位が細かいと速度面で不利になりやすい
- 標準入出力ベースだとエラー処理やログ管理が雑になりやすい
- Python 側と Node 側の責務境界が曖昧になる恐れがある

#### 評価

| 観点 | 評価 |
| --- | --- |
| 初期導入コスト | 低い |
| 長期保守性 | 中程度 |
| データ生成との相性 | 小規模なら可、大量生成では課題あり |
| テストしやすさ | 最小検証には向くが運用では工夫が必要 |

### 候補C: Node 側に薄いブリッジ層を作り、JSON 入出力でやり取りする

#### 概要

kobalab CPU 本体は Node / JavaScript 側に保持しつつ、Python から扱いやすい薄いブリッジ層を挟み、JSON ベースで state 相当情報を渡し、推奨手や補足情報を受け取る方式である。

#### 利点

- 既存ロジックを温存しやすい
- 入出力仕様を明文化しやすい
- デバッグログやテストケースを JSON ベースで蓄積しやすい
- 後から Python 移植する場合も、ブリッジ仕様が比較基準として残る
- data-format.md で想定している JSON 系データと相性が良い

#### 欠点

- ブリッジ層の設計と保守が必要
- state / action 変換仕様を丁寧に詰める必要がある
- 実行性能はネイティブ統合より不利な可能性がある

#### 評価

| 観点 | 評価 |
| --- | --- |
| 初期導入コスト | 中程度 |
| 長期保守性 | 比較的高い |
| データ生成との相性 | 良い |
| テストしやすさ | 高い |

### 候補D: 必要ならローカル API 化する

#### 概要

Node 側をローカルサービスとして起動し、HTTP などのローカル API で Python と通信する方式である。

#### 利点

- 長寿命プロセスとして管理しやすい
- 並列呼び出しや複数クライアント接続に発展しやすい
- API 契約を明文化しやすい

#### 欠点

- 初期段階では構成がやや重い
- サービス管理やタイムアウト設計が必要
- ローカル専用開発では過剰設計になる可能性がある

#### 評価

| 観点 | 評価 |
| --- | --- |
| 初期導入コスト | 中〜高 |
| 長期保守性 | 条件次第で高い |
| データ生成との相性 | 良い可能性がある |
| テストしやすさ | API テストはしやすいが全体はやや重い |

## 4. 当面の比較観点

接続方式を判断する際は、以下の軸で比較する。

| 比較観点 | 見るポイント |
| --- | --- |
| 実装コスト | 最初の疎通までに必要な作業量 |
| 既存ロジックとの一致性 | 既存 kobalab の意思決定をどれだけ忠実に再利用できるか |
| 保守性 | 将来の修正、検証、置き換えがしやすいか |
| デバッグしやすさ | ログ、再現、問い合わせ内容の追跡がしやすいか |
| データ生成速度 | 大量対戦や教師手生成に耐えられるか |
| Python エンジンとのつなぎやすさ | `engine-spec.md` の state / action 設計と繋げやすいか |
| 将来的な移行のしやすさ | Python 移植や高速化に進む際の足場になるか |

### この段階で特に重視すること

- 最速であることより、正しく比較できること
- 移植工数の過小評価をしないこと
- 外部呼び出しの運用負荷も過小評価しないこと
- state / action の受け渡し仕様を安定して検証できること

## 5. データ受け渡しの最小単位

### Python 側から渡す必要があるもの

少なくとも以下の情報が必要と考える。

| 項目 | 内容 | 備考 |
| --- | --- | --- |
| 局面表現 | kobalab が意思決定するために必要な state 情報 | 最終形式は未確定 |
| 手番 | 現在行動するプレイヤー | 必須 |
| ルール関連メタ情報 | 必要ならルール差分や設定 | 未確定 |
| 合法手情報 | legal actions を渡すか、kobalab 側で再計算するか | 要検討 |
| 問い合わせ ID | ログ追跡や再現用の識別子 | 推奨 |

### kobalab 側から受け取りたいもの

| 項目 | 内容 | 重要度 |
| --- | --- | --- |
| 推奨手 | 選択された action | 必須 |
| 応答ステータス | 成功 / 失敗 / unsupported など | 必須 |
| 補足メタ情報 | 必要なら候補手、評価値、理由、処理時間 | 任意 |
| エラー情報 | 失敗理由、入力不整合情報 | 重要 |

### 合法手情報の扱い

合法手については、以下の選択肢がある。

- Python エンジンが列挙した legal actions を渡す
- kobalab 側で独自に合法手を計算する
- 両方を持ち、検証用に比較できるようにする

初期段階では、比較検証の観点から、少なくとも Python 側と kobalab 側の不一致を検出できる余地を持たせることが望ましい。

### 受け渡し形式の考え方

- 人間可読であること
- ログ保存しやすいこと
- 異常系でも内容を確認しやすいこと
- `data-format.md` で想定している JSON 系保存形式と相性が良いこと

そのため、初期は JSON 互換構造を第一候補として考える。

## 6. 教師役として使う場合の要件

### 問い合わせ単位

教師役としては、1 局面ごとに問い合わせる方式が基本候補である。

理由:

- 模倣学習用サンプルは局面単位で作る前提と相性が良い
- `source_game_id` や `ply_index` と紐付けやすい
- 問い合わせ失敗時に局所的な再試行がしやすい

一方で、1 対局まとめて問い合わせる方式は、初期段階では柔軟性が低く、障害解析もしにくい可能性がある。

### 教師手の保存

- 生成した教師手は、学習用サンプルの `target_action` として保存する想定である
- 必要に応じて、人間可読表現や元問い合わせ ID も持てると望ましい
- 同一局面に対して同じ教師手が返るかを追跡できる形が望ましい

### 再現性

- 同じ局面に対して同じ結果が返ることが重要である
- ランダム性がある場合は、その有無と seed 管理方法を明示する必要がある
- 初期段階では、可能なら deterministic な利用を優先したい

### 将来ランダム性を持たせる場合

- 1 手の教師だけでなく、分布や複数候補を使いたくなる可能性はある
- ただし、初期段階では deterministic な単一推奨手で十分かをまず確認する

## 7. 対戦相手として使う場合の要件

### プレイヤーとしての扱いやすさ

- Python エンジン上の他プレイヤーと同列に扱えることが望ましい
- 具体的には、`state -> action` という最小インターフェースに乗せやすいことが重要である
- これにより random、学習済みモデル、kobalab を同じ対戦ループに入れやすくなる

### ログの取りやすさ

- 各手ごとの問い合わせと応答を追跡できること
- タイムアウト、エラー、unsupported action などが残ること
- `game_id` と `ply_index` をもとに後から再現しやすいこと

### タイムアウトや異常応答時の扱い

- タイムアウトの定義を持つ
- 異常応答を対局全体の失敗にするか、当該手の失敗にするかは未確定
- 初期段階では、異常を黙って補正せず、ログと失敗理由を残す方針が望ましい

### ベンチマーク用途

- 学習モデルと比較する際は、設定条件を固定しやすいことが重要である
- 先手後手、試合数、seed 条件、タイムアウト条件を明確に記録する必要がある

## 8. テスト観点

少なくとも以下の観点で検証したい。

| テスト観点 | 確認内容 |
| --- | --- |
| 同一局面での一貫性 | 同じ入力に対して同じ応答が返るか |
| 変換の完全性 | Python -> bridge -> kobalab の変換で情報欠落がないか |
| 既知局面応答 | 既知局面で期待手が返るか |
| 連続対戦安定性 | 複数対局を連続実行しても破綻しないか |
| 異常系処理 | 不正入力、タイムアウト、応答不正時の扱いが明確か |
| ログ追跡性 | 失敗時に原因調査できるログが残るか |

### 特に重要なテスト

- 同一局面での deterministic 性確認
- Python エンジンの legal_actions と kobalab の応答整合
- 既知局面を使った回帰テスト
- 連続実行時のメモリリークやプロセス残留の有無

## 9. 当面の第一候補

現時点で最も現実的な暫定第一候補は、Node 側の既存実装を温存しつつ、薄い JSON ブリッジ層で接続する方式である。すなわち、候補Cを第一候補とする。

### この方針を第一候補とする理由

- 既存ロジックとの一致性を保ちやすい
- Python 移植ほど初期コストが重くない
- 単純な subprocess 直呼びより、入出力契約を明文化しやすい
- JSON ベースでログ、テスト、再現ケースを残しやすい
- 将来 Python 移植する場合も、橋渡し仕様そのものが比較基準として活きる

### ただし断定しない点

- 大量データ生成時の速度が十分かは未検証である
- kobalab 側の既存構造次第では、CLI 直呼びの方が先に試しやすい可能性もある
- 長期的には Python 移植の必要性が高まる可能性がある

したがって、当面は「Node 温存 + JSON ブリッジ」を軸に小さく疎通確認し、必要性が高まった段階で Python 移植やローカル API 化を再評価するのが自然である。

## 10. 未確定事項

現時点で未確定または要確認の事項は以下の通りである。

- kobalab 側ロジックの再利用粒度
- 呼び出し方式の最終決定
- state 受け渡し形式の最終決定
- action 受け渡し形式の最終決定
- legal action をどちら側で持つか
- 評価値や候補手などの付加情報を受け取るか
- deterministic 利用をどこまで保証できるか
- 並列実行や高速化が必要になる時期
- 長期的に Python 移植するかどうか

## 11. 次アクション

### 先に確認すべきこと

- kobalab CPU の既存入出力形態
- 局面入力に必要な最小情報
- 推奨手以外に取得可能な情報の有無
- deterministic 性の有無

### 小さく試すなら最初にやること

- 単一局面を入力し、推奨手を 1 つ返す最小 I/O を確認する
- 入力局面と出力手を JSON 互換形式で記録する
- 同一局面に対する複数回問い合わせで応答が安定するか確認する
- エラー時の返却形式を観察する

### 切り出し候補文書

- `docs/bridge-interface.md`
  Python 側と Node 側の最小 I/O 契約
- `docs/kobalab-io-sample.md`
  局面入力と応答出力のサンプル集
- `docs/kobalab-test-plan.md`
  疎通確認と回帰テストの観点整理

本書は初版であり、最終方式を決めるものではない。ただし、初期段階では「正しく比較できること」「テストしやすいこと」「データ生成に使いやすいこと」を優先し、既存ロジックを温存しながら薄いブリッジで接続する方向を第一候補として扱う。
