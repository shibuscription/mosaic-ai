# MOSAIC 座標系整理メモ v0

## 1. 文書の目的

本書は、MOSAIC 学習型 CPU プロジェクトにおける通常 2人用 7×7 の盤面座標系を整理するための文書である。目的は、Python エンジン実装時に、盤面位置、上層位置、最小正方形、走査順、ログ出力の扱いで迷いを減らすことにある。

`docs/game-rules-standard.md` は通常 2人用 7×7 のルール一次資料、`docs/engine-spec.md` はエンジン責務整理、`docs/action-encoding.md` は action 表現整理、`docs/sample-data.md` は具体例集である。本書は、それらを横断して共通に使う座標の前提を独立文書として明確化する役割を持つ。

座標系を独立文書に切り出す理由は、同じ盤面位置が state、action、legal_actions、最小正方形判定、bridge request、fixture のすべてに関係するためである。座標系の認識がぶれると、実装やテストの差分原因が追いにくくなる。

## 2. 座標系の基本方針

### 基本方針

- 盤面位置は `level / x / y` を持つ 3 次元的な表現で扱う
- 各 `level` は 2 次元格子として扱う
- `level = 0` を最下層とする
- `x` は左から右へ増加する
- `y` は上から下へ増加する
- 2 次元格子の原点は左上とする

### この方針の意図

- JSON やログに落としやすい
- 2×2 最小正方形を左上基準で扱いやすい
- 「左上から右下へ」の deterministic 走査順と整合しやすい

### 座標の見方

| 軸 | 意味 | 増加方向 |
| --- | --- | --- |
| `level` | 層番号 | 下から上 |
| `x` | 横方向 | 左から右 |
| `y` | 縦方向 | 上から下 |

## 3. 各 level のサイズ

### 基本

- 最下層 `level = 0` の一辺サイズは 7
- 上に 1 層上がるごとに一辺サイズは 1 ずつ小さくなる

### 一般式

通常 2人用 7×7 では、`level = L` の一辺サイズを以下で扱う。

- `size(L) = 7 - L`

### 各 level のサイズ表

| level | 一辺サイズ | 座標範囲 |
| --- | --- | --- |
| 0 | 7 | `0 <= x,y <= 6` |
| 1 | 6 | `0 <= x,y <= 5` |
| 2 | 5 | `0 <= x,y <= 4` |
| 3 | 4 | `0 <= x,y <= 3` |
| 4 | 3 | `0 <= x,y <= 2` |
| 5 | 2 | `0 <= x,y <= 1` |
| 6 | 1 | `0 <= x,y <= 0` |

### 最上段

- 最上段は `level = 6`、サイズ 1×1 と考える
- これより上の層は存在しない

## 4. 位置の表現案

### 基本表現

1 つの位置は `(level, x, y)` で表す。

例:

- `(0, 0, 0)` は最下層左上
- `(0, 6, 6)` は最下層右下
- `(1, 0, 0)` は 1 層上の左上
- `(6, 0, 0)` は最上段の唯一の位置

### JSON での表現例

```json
{
  "level": 0,
  "x": 3,
  "y": 3
}
```

### ログでの表現例

- `L0(3,3)`
- `level=0,x=3,y=3`

### 人間可読性とのバランス

- 内部や JSON では `level/x/y` の明示的フィールドを優先する
- デバッグ表示では `L0(3,3)` のような短縮表記を補助的に使ってもよい
- ただし、短縮表記だけを唯一の表現にしない方が安全である

## 5. 最小正方形（2×2）の表現

### 基本定義

各層の最小正方形は、その層の隣接する 2×2 の 4 点で構成される。

### 左上基準の表し方

最小正方形は、その左上の位置 `(level, x, y)` を基準に表すのが扱いやすい。

そのとき、対応する 4 点は以下になる。

- `(level, x, y)`
- `(level, x+1, y)`
- `(level, x, y+1)`
- `(level, x+1, y+1)`

### 最下層 7×7 での数え方

`level = 0` の最下層では、左上基準座標の範囲は以下になる。

- `0 <= x <= 5`
- `0 <= y <= 5`

したがって、候補数は `6 × 6 = 36` 個になる。

### 一般化

一辺サイズが `n` の層では、最小正方形候補数は `(n-1) × (n-1)` 個となる。`level = L` の場合は `size(L) = 7 - L` なので、

- 候補数 = `(size(L) - 1) × (size(L) - 1)`

と整理できる。

## 6. 上層位置の導出

### 基本

ある層 `level = L` の最小正方形の上には、`level = L + 1` の 1 つの位置が対応する。

### 対応関係

左上基準の最小正方形が `(L, x, y)` のとき、その上層位置は

- `(L + 1, x, y)`

で表す。

### 例

- `level 0` の 2×2 左上 `(0, 2, 3)` の上は `(1, 2, 3)`
- `level 3` の 2×2 左上 `(3, 1, 0)` の上は `(4, 1, 0)`

### 実装上の利点

- 最小正方形と上層位置の対応が単純になる
- 自動配置判定と連鎖処理で使いやすい
- 2×2 の左上基準と上層位置の座標を一致させられる

### 下層から上層への写像としての見方

- 入力: 最小正方形の左上位置 `(L, x, y)`
- 出力: その上に置かれる候補位置 `(L+1, x, y)`

この写像を固定することで、最小正方形判定と自動配置候補生成の整合が取りやすい。

## 7. 走査順序の考え方

### 基本方針

再現性のため、座標系と整合した deterministic な走査順序を採用する。

### 暫定方針

- 下層から上層へ
- 同一層では左上から右下へ

### 同一層での意味

同一層では、以下の順序を想定する。

1. `y` の昇順
2. 同じ `y` の中で `x` の昇順

すなわち、行優先で左上から右下へ走査する。

### 座標系との関係

- 原点を左上に置く
- `x` を右方向、`y` を下方向とする

この前提により、「左上から右下へ」が `y` 昇順、次に `x` 昇順という自然な形で定義できる。

### 利用箇所

- legal_actions の列挙順
- 最小正方形候補の走査順
- 自動配置候補の解決順
- 連鎖処理順

## 8. 他文書への影響

### state 表現

- 各コマ配置を `(level, x, y)` で参照しやすくする必要がある
- `board` の内部構造がどうであっても、最終的にこの座標系に対応できることが重要である

### action 表現

- `place` 系 action は、どの位置に置くかを `(level, x, y)` または同等の JSON で持つと整理しやすい
- 自動配置結果も同じ座標体系で表現できる

### sample-data の例

- 位置例や bridge 例に `level` を含めるかどうかを揃える必要がある
- 現在の簡略例では `level` が省略されている箇所があるため、今後揃える候補になる

### bridge request / response

- bridge 交換表現でも、位置情報は `level/x/y` ベースに寄せると整合が取りやすい
- Python 内部表現と bridge 交換表現を分ける場合も、座標軸の意味は揃えるべきである

### テスト fixture

- 既知局面や最小正方形判定テストを座標指定で書きやすくなる
- 走査順や連鎖順の期待値を表現しやすくなる

## 9. 現時点での暫定方針

通常 2人用 7×7 に対して、現時点で最も実装しやすい暫定方針は以下である。

- 位置は `(level, x, y)` で表す
- `level = 0` を最下層とする
- 原点は各層の左上とする
- `x` は左から右、`y` は上から下とする
- 最小正方形は左上基準 `(level, x, y)` で表す
- その上層位置は `(level + 1, x, y)` とする
- 走査順は `level` 昇順、同一層では `y` 昇順、次に `x` 昇順とする

この方針は、ルール文書の「下層から上層へ」「同一層では左上から右下へ」という暫定処理順と整合しやすく、実装・ログ・fixture のいずれでも扱いやすい。
